# plugins/prestashop — Compass

PrestaShop module: Frak SDK injection, order webhook sender, admin config. PHP 8.1+, PrestaShop 8.1+. Module slug `frakintegration`. Installed by uploading the build zip via the PrestaShop admin (no composer install on merchant side — vendor/ ships inside the zip).

## Quick Commands
```bash
composer install                             # Deps (from this dir)
./build.sh                                   # Package module zip for release (dist/)
vendor/bin/phpunit                           # Unit tests (test/Unit)
vendor/bin/phpstan analyse                   # Static analysis (real PS surface via prestashop/php-dev-tools)
vendor/bin/phpcs --standard=phpcs.xml.dist   # Style (PSR-12 baseline)
```

## Key Files
- `frakintegration.php` — module bootstrap (`class FrakIntegration extends Module`). Thin router: every `hookXxx()` is a one-line delegator to a helper in `classes/`. PrestaShop discovers hooks by reflection so the methods MUST live on the Module class — but everything else (lifecycle, hook bodies, render glue, Smarty registration) is decomposed.
- `config.xml` — PrestaShop module manifest (display name, version, tab) — version is rewritten by `build.sh`.
- `composer.json` — canonical version source (`"version"` field). Runtime deps: `symfony/http-client`, `symfony/cache`, `symfony/lock`, `doctrine/dbal`. Dev tooling: phpunit / phpstan + `prestashop/php-dev-tools` / phpcs. The `composer analyse` script wraps PHPStan: it auto-fetches PrestaShop 8.2.6 sources into `.cache/prestashop-core/` (gitignored, runs `composer install --no-dev` inside the PS clone), then runs phpstan with `_PS_ROOT_DIR_` pointed at it so the analyser sees the real `Module` / `Configuration` / `Tools` / `Db` / `Order` / … surface.
- `classes/FrakUtils.php` — host normaliser (`currentHost()`); mirrors WordPress `Frak_Utils::current_host()`.
- `classes/FrakMerchantResolver.php` — resolves the Frak merchant UUID from the shop domain via `GET /user/merchant/resolve` (Symfony HttpClient), with an eternal cache and 5-min negative cache backed by the Symfony Cache pool. Mirrors WordPress `Frak_Merchant`.
- `classes/FrakWebhookHelper.php` — HMAC-SHA256 signed `POST /ext/merchant/{id}/webhook/custom` sender, built on Symfony HttpClient. Signature is `base64_encode(hash_hmac('sha256', $body, $secret, true))` (raw bytes, base64-encoded — NOT hex), matching `validateBodyHmac` in `services/backend/src/utils/bodyHmac.ts`. Tight 5 s timeout. Single-shot via `request()`, batch via `client->stream()` for HTTP/2-multiplexed parallel dispatch. Delivery logs go through `FrakLogger` (which forwards errors to `PrestaShopLogger`).
- `classes/FrakComponentRenderer.php` — emits `<frak-button-share>`, `<frak-post-purchase>`, `<frak-banner>` markup. Mirrors WordPress `Frak_Component_Renderer`; PrestaShop bootstrap classes substitute for WP's `wp-element-button` in the `buttonStyle` preset table. Public `snakeKeysToCamel()` helper feeds the Smarty function plugins so templates can use snake_case attribute keys.
- `classes/FrakInstaller.php` — install / uninstall lifecycle (CORE_HOOKS const, hook chain, `sql/install.php` + `sql/uninstall.php` includes, `FrakDb::createInfrastructureTables()`, `Configuration` seeding, cron token provisioning, `FrakPlacementRegistry::seedDefaults()`). Symmetric uninstall. Called from `FrakIntegration::install()` / `uninstall()` after `parent::install()`/`uninstall()`.
- `classes/FrakFrontend.php` — front-office head + asset wiring. `head()` emits the `dns-prefetch` / `preconnect` resource hints + inline `window.FrakSetup` config block (no Smarty). `setMedia($context)` registers the SDK script through PrestaShop's native asset manager (`registerJavascript`, `defer`, bottom, priority 200). Mirrors WordPress's `Frak_Frontend`.
- `classes/FrakOrderHooks.php` — order-surface hook bodies. `onStatusUpdate($params)` maps PS order states to Frak webhook statuses + sync-attempt-then-retry-queue dispatch. `renderPurchaseTracker($order)` emits the always-on `trackPurchaseStatus` script (fired regardless of placement state). `renderPostPurchase($module, $order, $placement)` resolves order data via `FrakOrderResolver`, emits `<frak-post-purchase>` markup, and wraps it through the theme-overridable Smarty partial. Mirrors WordPress's `Frak_WooCommerce`.
- `classes/FrakDisplayDispatcher.php` — placement-driven `display*` hook router. `dispatch($module, $hook, $params)` iterates `FrakPlacementRegistry::forHook($hook)`, gates each on `isEnabled()`, routes the Banner / Share / Post-Purchase components to their respective renderers. Replaces the legacy `FrakIntegration::dispatchHook()` private method.
- `classes/FrakSmartyPlugins.php` — front-office `{frak_*}` Smarty function plugin registration + handlers. `register($context)` is idempotent across module instantiations within a request via a static flag; `banner()` / `shareButton()` / `postPurchase()` are the three static handlers (snake_case → camelCase via `FrakComponentRenderer::snakeKeysToCamel`). Mirrors WordPress's `Frak_Shortcodes`.
- `controllers/` — admin + front controllers.
- `sql/install.php` / `sql/uninstall.php` — schema lifecycle (CREATE / DROP) for `frak_webhook_queue`. Loaded from `install()` / `uninstall()` via `include`. Adding a new table = one entry in each file.
- `upgrade/install-X.Y.Z.php` — PrestaShop-native upgrade scripts. Auto-discovered by filename; PrestaShop runs `upgrade_module_X_Y_Z($module)` whenever the filesystem `config.xml` version exceeds the DB `ps_module.version`. Replaces the homegrown `ensureSettingsMigrated()` pattern — `ps_module.version` is the single source of truth for migration state.
- `override/` — empty placeholder directory (just an `index.php` security stub). PrestaShop expects it on every module.
- `classes/FrakPlacementRegistry.php` — single source of truth mapping each component × hook tuple to a `FRAK_PLACEMENT_*` Configuration toggle. `PLACEMENTS` is the authoritative list; `distinctHooks()` drives install/uninstall registration, `forHook()` drives the dispatch loop, `seedDefaults()` runs from install + migrator, `clearAll()` runs from uninstall.
- `classes/FrakWebhookQueue.php` — `{prefix}frak_webhook_queue` table accessor (enqueue, due, markSuccess, markFailure, **stats**) built on Doctrine DBAL. Schema lives in `sql/install.php` / `sql/uninstall.php`. Backoff schedule mirrors Magento: 5 m/15 m/1 h/6 h/24 h, max 5 attempts. `stats()` aggregates `(state, count, oldest_pending_at)` plus the latest non-empty `last_error` for the admin observability panel — two indexed queries, safe per render.
- `classes/FrakDb.php` — shared infrastructure factory. Owns the single Doctrine DBAL connection used by every Frak class that touches MySQL, plus the Symfony Cache pool (`DoctrineDbalAdapter` against `{prefix}frak_cache_items`) and Symfony Lock factory (`DoctrineDbalStore` against `{prefix}frak_lock_keys`) built on top of that connection. `createInfrastructureTables()` / `dropInfrastructureTables()` run from install / uninstall to bootstrap the cache + lock schemas.
- `classes/FrakWebhookCron.php` — cron drainer (`run()` selects due rows, calls WebhookHelper, transitions state).
- `controllers/admin/AdminFrakIntegrationController.php` — admin form. Single “Save Settings” submit covers brand + webhook secret + placement toggles. Surfaces the cron URL (`{base}/index.php?fc=module&module=frakintegration&controller=cron&token=<FRAK_CRON_TOKEN>`), the queue health snapshot, and a “Drain queue now” button (synchronous `FrakWebhookCron::run()`) so the merchant can flush a backlog without waiting for the next cron tick.
- `controllers/front/cron.php` — token-guarded (`hash_equals`) cron front controller; drains the retry queue per tick.
- `views/` — Smarty templates (`hook/head.tpl`, `admin/configure.tpl`).
- `test/Unit/` — PHPUnit tests.
- `test/docker-compose.yaml` — manual integration env (PrestaShop + MySQL on `localhost:8080`).
- `dist/` — build output (zip lives here, gitignored).

## Non-Obvious Patterns
- **Backend-driven configuration** — only `metadata.{name,logoUrl}` is injected on `window.FrakSetup`. Everything else (i18n, language, modal config, walletUrl, domain, share-button copy/style) is resolved by the SDK against `business.frak.id` once the merchant is registered. Legacy v0.0.4 rows (`FRAK_MODAL_*`, `FRAK_SHARING_BUTTON_*`, `FRAK_FLOATING_BUTTON_*`, `FRAK_WEBHOOK_LOGS`) are wiped by `upgrade/install-1.0.1.php` on the first PrestaShop upgrade dispatch — see the audit baseline at `https://github.com/frak-id/prestashop-plugin/blob/v0.0.4/frakintegration.php`.
- **Webhook secret is pasted, not generated** — the merchant copies the secret from `business.frak.id` (Merchant → Purchase Tracker → PrestaShop) into the admin form. There is no "Generate" button; the dashboard owns the canonical value. Mirrors the WordPress plugin's flow.
- **Async webhook with retry queue** — `hookActionOrderStatusPostUpdate` (post-commit, NOT pre-commit `actionOrderStatusUpdate`) attempts a synchronous send with 5 s request / 3 s connect timeouts; any failure (network, non-2xx, exception) is enqueued into `frak_webhook_queue` for cron retry with exponential backoff (5 m/15 m/1 h/6 h/24 h, max 5 attempts). Order-status transitions never block the merchant for more than a few seconds. Idempotent: backend dedupes by `(merchantId, externalId, status)` so re-deliveries are safe.
- **Webhook signature is base64, not hex** — `base64_encode(hash_hmac('sha256', $body, $secret, true))`. The default hash_hmac() return value is hex; sending hex produces a 64-byte buffer after base64-decoding on the backend (vs the expected 32-byte raw digest) and silently fails verification. Magento and WooCommerce sister plugins use the same base64 contract.
- **Webhook URL is `/webhook/custom`** — reuses the existing `customWebhook` Elysia route on the backend (DTO matches what we send: `id`, `customerId`, `status`, `token`, `currency`, `totalPrice`, `items`). No platform-specific `/webhook/prestashop` route exists.
- **Cron token in URL, not session** — the cron front controller is hit by external cron (`ps_cronjobs` or server cron) outside any admin session, so authentication is a 64-char hex `FRAK_CRON_TOKEN` query param compared via `hash_equals`. The full URL is rendered on the admin configuration page for copy-paste. Rotating the token requires deleting `FRAK_CRON_TOKEN` and reinstalling.
- **Component rendering goes through `FrakComponentRenderer`** — every hook surface dispatches via `FrakDisplayDispatcher::dispatch($this, $hook, $params)` which iterates `FrakPlacementRegistry::forHook($hook)`, gates each on `isEnabled()`, and routes to `FrakComponentRenderer::shareButton(...)` / `::banner(...)` or `FrakOrderHooks::renderPostPurchase(...)`. Adding a placement is a one-line edit in `FrakPlacementRegistry::PLACEMENTS` plus a matching `hookXxx()` callback that delegates to `FrakDisplayDispatcher::dispatch()`.
- **Thin module bootstrap, surface-based helpers** — `frakintegration.php` is intentionally a 150-line router. `hookXxx()` methods MUST stay on the Module class (PrestaShop's reflection-based discovery), but their bodies all delegate to `FrakInstaller` / `FrakFrontend` / `FrakOrderHooks` / `FrakDisplayDispatcher` / `FrakSmartyPlugins`. Mirrors WordPress's split between `frak-integration.php` (entry) and the per-surface classes. When adding new behaviour: extend the relevant helper, NOT the bootstrap. The Module class only ever grows on PrestaShop hook signature changes.
- **Smarty function plugins (`{frak_banner}`, `{frak_share_button}`, `{frak_post_purchase}`)** — registration lives in `classes/FrakSmartyPlugins::register($context)`, called from `FrakIntegration::__construct()` on every front-office module instantiation (cheap, idempotent: static flag + `unregisterPlugin()` first then `registerPlugin()`). Lets merchants drop components into any `.tpl` file or CMS page without forking the module. Snake_case attribute keys are normalised to camelCase via `FrakComponentRenderer::snakeKeysToCamel()` so templates read naturally (`{frak_banner referral_title="..."}`). Mirrors the WordPress `[frak_*]` shortcode contract.
- **Placement toggles persist via hidden `__present` markers** — PrestaShop’s `<input type="checkbox">` is omitted from `$_POST` when unchecked, which would prevent merchants from ever turning a placement off. The template emits a hidden `<input name="FRAK_PLACEMENT_FOO__present" value="1">` next to each checkbox; the controller treats “`__present` set + checkbox absent” as “write 0” and “`__present` absent” as “leave alone”. Pattern lifted from WordPress’ Settings API checkbox-handling convention.
- **Single Save button, separate maintenance buttons** — brand + webhook + placements share one `<form>` so config saves are atomic (one round-trip, single confirmation message). Refresh Merchant + Drain queue live in their own small `<form>`s under a Maintenance panel because they are stateful operations that should not be triggered by every config save.
- **Three version surfaces, one source**: `composer.json#version` is canonical. `build.sh` propagates to `config.xml` (`<version>`) and `frakintegration.php` (`$this->version`) inside the staged build dir only — the source tree carries the last-released values, the released zip always reflects the resolved version. Bump composer.json (the release workflow does this for you), never edit the other two by hand for releases.
- **Vendor ships in the zip**: PrestaShop merchants install via admin upload, not composer. `build.sh` runs `composer install --no-dev --optimize-autoloader` before staging, and `.distignore` keeps `composer.json` / `composer.lock` out (so composer can't accidentally re-run on the merchant side).
- **`override/` is committed empty** — PrestaShop expects the directory on every module. The source tree carries an `override/index.php` security stub so the layout mirrors the released zip without requiring `build.sh` to mkdir it.
- **PSR-12, not PrestaShop's own ruleset** — phpcs uses the PSR-12 baseline shipped with phpcs. The `prestashop/php-dev-tools` package is required (dev) for the PHPStan extension (`vendor/prestashop/php-dev-tools/phpstan/ps-module-extension.neon`); the cs-fixer config it ships is unused here.
- **PHPStan analyses against real PrestaShop 8.2.6 sources** — the `composer analyse` script clones `PrestaShop/PrestaShop@8.2.6` into `.cache/prestashop-core/` (idempotent) and runs `composer install --no-dev` inside that clone so the PS composer autoloader resolves Symfony / Smarty / Doctrine deps that PS classes touch. `_PS_ROOT_DIR_` then points the `prestashop/php-dev-tools` PHPStan bootstrap at that checkout. Replaces the previous hand-rolled `phpstan-stubs.php` (~570 lines of PS class stubs) — drift between the stubs and the real PS API can no longer happen because the analyser reads PS straight from source. The pinned tag (8.2.6, the latest 8.2.x patch) sits one minor above our declared minimum (PrestaShop 8.1+) so we catch class-surface changes that landed in 8.2 before merchants on the latest stable hit them. CI caches the clone via `actions/cache` keyed on `prestashop-core-8.2.6`; bump the tag in `composer.json#ps-core:fetch` (and the cache key in `.github/workflows/php-plugins.yaml`) to roll forward.
- **`./build.sh` produces `dist/frakintegration-<version>.zip`** — that zip is the artifact uploaded to merchants via the PrestaShop admin uploader.
- **One shared DBAL connection for all infra** — Cache (`DoctrineDbalAdapter`), Lock (`DoctrineDbalStore`), and the webhook queue all hit the same `FrakDb::connection()`, so the per-request DB connection count stays at PS native + 1. The connection params are built from `_DB_SERVER_` / `_DB_NAME_` / `_DB_USER_` / `_DB_PASSWD_` rather than reusing `Db::getInstance()->getLink()` because the latter returns mysqli on shops without `PS_USE_PDO`, and we need a guaranteed PDO-backed handle for DBAL.

## Release Flow
- CI: `.github/workflows/php-plugins.yaml` runs `cs` + `analyse` + `test` on every push touching `plugins/**`.
- Release: dispatch `.github/workflows/release-prestashop.yml` with the new version → opens a `release/prestashop-<version>` PR (bumps `composer.json`, promotes `[Unreleased]` in `CHANGELOG.md`). Merging the PR with the `release:prestashop` label triggers the publish job, which tags `prestashop-<version>`, builds the zip, and creates a GitHub release.

## Anti-Patterns
Hand-editing `config.xml` / `frakintegration.php` versions for a release (let `build.sh` propagate) · committing `vendor/` (gitignored) · adding bundle/zip scripts outside `build.sh` (single source) · running fire-and-forget HTTP for webhooks from the order hook without a queue fallback (use the WebhookHelper + Queue + Cron triad) · using `actionOrderStatusUpdate` instead of `actionOrderStatusPostUpdate` (pre-commit hook races under multistore / high load) · sending the HMAC signature as hex (`hash_hmac('sha256', $body, $secret)` returns hex by default — the backend decodes as base64; ALWAYS pass the third arg `true` for raw and `base64_encode` the result) · reintroducing per-merchant share-button / modal-i18n config (live in `business.frak.id`) · reintroducing the Configuration-backed webhook log ring (use `PrestaShopLogger`) · generating the webhook secret locally (paste from dashboard) · storing the retry queue in `Configuration` (autoloads on every request, see `FrakWebhookQueue` rationale) · hard-coding placement hooks in `frakintegration.php` instead of registering them through `FrakPlacementRegistry` (the install/uninstall/migrator/dispatch paths read from the same list — hard-coding any one of them re-introduces drift) · omitting the hidden `__present` marker next to a placement checkbox (unchecked checkboxes never submit; without the marker the merchant can never turn the placement off) · putting Refresh Merchant or Drain Queue inside the main settings `<form>` (they fire on every Save submit and would refresh / drain unintentionally) · spinning up a fresh PDO / DBAL connection inside individual classes when `FrakDb::connection()` is the single source of truth (would multiply per-request connection count) · raw cURL or `Tools::file_get_contents` for outbound HTTP (use `\Symfony\Component\HttpClient\HttpClient::create()` so timeouts + connection pooling + HTTP/2 multiplexing come for free).
